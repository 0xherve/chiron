# Database Schema

## Tables

**Users**
- `email`
- `name`
- `avatarUrl` (optional)
- `createdAt`, `updatedAt`

**Rubrics**
- `userId` (FK → Users)
- `definition` (JSON)
- `createdAt`, `updatedAt`

**Threads**
- `userId` (FK → Users)
- `rubricId` (FK → Rubrics) — set once at creation, immutable
- `title` (optional, AI-generated or user-edited)
- `createdAt`, `updatedAt`

**Documents**
- `threadId` (FK → Threads)
- `docRef` (R2 object key)
- `wordCount`
- `createdAt`

**Gradings**
- `threadId` (FK → Threads)
- `rubricId` (FK → Rubrics)
- `documentId` (FK → Documents)
- `status` (`pending` | `completed` | `failed`)
- `overallScore` (optional)
- `summary` (optional)
- `createdAt`

**Messages**
- `threadId` (FK → Threads)
- `role` (`user` | `assistant` | `system`)
- `kind` (`text` | `document_upload` | `system`)
- `content`
- `createdAt`

## Relationships & Rules

- A thread references one rubric; the rubric cannot be changed after thread creation.
- Every grading records the exact rubric and document used.
- Documents are stored in R2; only the reference and metadata live in the database.
- Gradings are stored separately from messages and appear in the right grading panel.
- The last graded document is derived from the most recent grading.
- Thread titles can be auto-generated by AI or manually set/updated by the user.
